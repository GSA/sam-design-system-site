Progressive enhancement

Progressive enhancement is a web development approach with iterative design at its heart.

The following are implemented in turn — focusing on the next only when it appears the first cannot satisfy the user need. By doing so you are able to leverage and present base functionality while maintaining the spirit of the web: Content, functionality, and code, which is open and accessible to anyone, anywhere so long as they are connected to a network, regardless of the technology used.

A potentially shippable increment in a short time is also afforded, because the content is made available first; the user experience and functionality of which is enhanced iteratively, over time based on user-need and feedback (the second value of the Manifesto for Agile Software Development and the first principle). Finally, it is easier to change and iterate when the codebase is small; thereby, allowing the possibility of rapid user feedback and adjustment before having too much code to change quickly and a “locked” architecture (the second and third principles of the Manifesto for Agile Software Development).

Content

As the saying goes, “Content is king.” Most web pages are, first and foremost, content — not functionality.

Determine:

  • What are you trying to communicate or convey?
  • What information should be displayed on a page?
  • How should the content be formatted and ordered?
  • And so on.

By considering these questions before writing code we ensure users get the information they need in a consumable way.

Semantic markup (structure)

HTML is the foundation of the web — without it, there is no page; therefore, everything else can be considered an extra.

The HTML specification is maintained by the World Wide Web Consortium to inform browser (user-agent) developers how to render HTML. Further, the HTML specification lets you know when elements should be used and where; this is what is meant by “semantic markup”.

Semantic markup aids users in reading and navigating the page and site. Assistive technologies use the HTML elements to aid users in navigating the content. Therefore, it is essential to choose elements based on their semantic correctness — not their visual appearance. For example, using h1 instead of bold or something else for the title of a page.

CSS and CSS3 (style)

The CSS specification is maintained by the World Wide Web Consortium and allows you to separate the structure of a page or site from its aesthetics.

CSS level 1 has been available since 1997. CSS level 3, introduced in 2014, allows for even more interactive capability and selector options to be handled via CSS (animations or alternating table row colors, for example); therefore, no longer requiring complex logic to generate different markup using some type of template system.

CSS allows you to create visually appealing layouts and designs to an entire site, without tying the presentation to the HTML itself…and it’s pretty amazing the flexibility you are afforded by HTML and CSS alone.

Note: CSS is built with a progressive enhancement approach, with CSS level 1 also being robust, flexible, and still in use today.

Note: We avoid modifying the base CSS generated by the US Web Design Standards as much as possible; opting to contribute directly to that project through issues and pull requests.

JavaScript (enhanced interactivity)

The JavaScript specification is maintained by ECMA International (not a World Wide Web Consortium specification).

JavaScript manipulates the underlying HTML string of a page and can also make calls to a server without moving the user to a new page. These interactive concepts could be difficult or impossible to do with HTML and CSS alone. Further, they only serve to update the HTML and style of a page. Therefore, JavaScript is not a replacement for the specifications of the World Wide Web Consortium, but an extension of, the web ecosystem itself.

Modular design

Modular design is the mindset connecting encapsulated units of functionality together will create a system whereby the whole is greater than the sum of the parts and allows for complex systems based on simple pieces.

This approach is at the heart of a micro-service architecture. By maximizing reusable modules and methods, you are able to create simple (or even complex) user interactions, which are easy to change in the future.

A module mindset, coupled with progressive enhancement, allows for iterative design and emergent architectures; in part, because modular design maintains small pieces of encapsulated functionality across the entire system. In other words, the pursuit of modules becomes a treasure hunt!

(Module Go! Gotta find ’em all. Too bad there’s not an app for that.)

As you build a page, the idea for a reusable concept emerges. Then it is just a matter of refactoring and re-engineering the code to extract the discovered module from its surroundings. Building another page, that uses the module from the first, may spark the idea for another module…and so on. (See also Don’t Repeat Yourself.)

Example

The idea for the UI kit emerged while building a prototype environment with a proof of concept for a single system feature.

  • Mark 1: Prototype environment setup using template engine from selected framework as a proof of concept.
  • Mark 2: Maximizing reuse caused the creation of multiple “partials” wherein an array of configuration options were passed in order to generate the HTML.
  • Mark 3: Eeach “partial” was extracted in the native language of the framework; thereby, allowing the library to be used independent of framework chosen by developers.

Behavior driven development

Behavior driven development was born from test driven development.

Test driven development is a development and testing practice whereby tests are written before production code is developed. This test will initially fail (red); at which point, the simplest code required to pass the test is written. The test should now pass (green); at which point, you refactor (make it easier for developers to read) and, possibly, re-engineer (an extension of refactoring, which also benefits the end-user).

Behavior driven development asks you to consider the end-user when writing tests: What should this thing do from a user perspective? If this happens, how should this thing respond? And so on.

Tests written for behavior driven development typically take the form of a complete sentence covering: the state of the system under test, the input or interaction from the user, and the expected result based on that interaction.

Testing the validity of the system is actually not the mindset here. Instead, the tests act as a safety net to allow you to feel relatively safe that the re-engineering of your code does not unexpectedly deviate from your expectations.

By leveraging this “red-green-refactor” pattern (the TDD waltz) you can feel relatively safe when you discover a new module (or change a design pattern) that the system still performs as expected while you strive for technical excellence (the ninth principle of the Manifesto for Agile Software Development.

Bandwidth and Processor Speeds

The Standards views mobile devices as first-class citizens with regard to how users interact with the web. It is important to consider bandwidth constraints, caps on data plans, and processor speeds. Mobile and tablet devices typically have less processing power and hardware resources available, which should also be taken into consideration when creating client-side scripts.

URLs

Secure by Default

There are two transfer protocols used to serve Web content: Hypertext Transfer Protocol (HTTP) and HTTP Secure (HTTPS). By default all uniform resource locators (URLs) should use HTTPS; an HTTP URL should redirect to the HTTPS equivalent.

No WWW

URLs should not require the use of “www”, as this represents a sub-domain and could lead to confusion if subdomains are implemented on the site. In other words, there is a question of whether the following are different sites or the same:

  • https://www.my.domainname.com
  • https://my.www.domainname.com
  • https://my.domainname.com

Human-friendly and readable

URLs should be case-insensitive; therefore, the following URL pairs should result in the display of the same content:

  • https://My.DomainName.com and
    https://my.domainname.com
  • https://my.DomainName.com/MainPage/SubPage and
    https://my.domainname.com/mainpage/subpage
  • https://my.domainname.com/AFileOnTheServer.pdf and
    https://my.domainname.com/afileontheserver.pdf

This allows communication and marketing collateral to be built in a more human-readable manner by using upper- and lower-case letters to separate words within URLs while supporting copy and paste of the URL into a browser.

Further, when developing page addresses and URLs a user should be able to gain a basic understanding of what they will see, for example:

  • https://domainname.com/mainpage/posts/descriptive-post-title
  • not https://domainname.com/?page=1&post=5

Feature toggles

A feature toggle attempts to solve the problems seen in feature branches and other complex branching strategies related to build features while still releasing software.

One way to create a feature while ensuring it does not get released to the public until it is ready is to create a feature branch within a version control system (like Git). The feature toggle can be set to look at the version number within a file, the environment flag in another, something else entirely, and any combination thereof. Therefore, allowing you to begin working on a feature while continuously updating the main deployment branch, with a great level of confidence that the feature will not be visible to end-users until it is ready.

For example, some times Facebook will make a new feature available, but only to users within a certain city, or with certain email domain names, or any other criteria they choose. This way, not all users see the new feature, and Facebook can receive feedback from the users who do.
